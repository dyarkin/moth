# Sync and Conflicts

Sync is the workflow that turns modules into real config files.

## The Full Workflow

`moth sync` does three things:

1. Compiles each module's templates into that module's `.compiled` directory.
2. Combines all module `.compiled` directories into the root `~/.moth/.compiled` directory.
3. Applies symlinks from real target paths to files in the root `.compiled` directory.

This means the generated files remain owned by Moth, while your system sees them at their normal locations.

## Compile

Compile one module:

```sh
moth compile git
```

Compile all modules:

```sh
moth compile --all
```

Compilation renders templates with variables, enabled presets, and snippets.

Output example:

```text
~/.moth/git/.compiled/.gitconfig
```

Compiling does not create or change target symlinks.

## Prepare

Prepare without applying symlinks:

```sh
moth sync --prepare-only
```

This writes the root compiled tree:

```text
~/.moth/.compiled
```

Use this when you want to inspect generated files before changing target paths.

Skip recompilation and use existing module `.compiled` directories:

```sh
moth sync --no-compile
```

During prepare, every non-hidden module must have a matching `moduleRoots` entry in `config.yaml`.

## Root Compiled Paths

The root `.compiled` tree mirrors absolute target paths.

Example config:

```yaml
moduleRoots:
  git: ~
```

Template:

```text
~/.moth/git/templates/.gitconfig
```

Prepared source path:

```text
~/.moth/.compiled/Users/<user>/.gitconfig
```

Target path:

```text
/Users/<user>/.gitconfig
```

The exact prefix depends on your operating system and home directory.

## Apply

Apply an already prepared root `.compiled` tree:

```sh
moth apply
```

`apply` creates symlinks at the target paths.

Example:

```text
~/.gitconfig -> ~/.moth/.compiled/Users/<user>/.gitconfig
```

Moth creates parent directories when needed. It creates symlinks to files, not symlinks to whole directories.

## Target State

Moth records managed symlinks in:

```text
~/.moth/.targets.local.yaml
```

This lets Moth remove obsolete symlinks later, but only when they still point to the source Moth previously managed.

Do not edit this file unless you are repairing local state manually.

## Module Output Conflicts

Two modules can target the same final file.

Allowed case:

- Zero or one `base` template for the target file.
- Any number of `fragment` templates for the target file.

Conflict case:

- More than one `base` template targets the same file.

Moth also reports a conflict when one compiled target path would need to be both a file and a directory.

Use template roles in `manifest.yaml` to make intentional composition explicit.

## Target Filesystem Conflicts

Moth is conservative.

It refuses to overwrite:

- Existing regular files.
- Existing directories at target file paths.
- Symlinks pointing somewhere unexpected.
- Previously managed symlinks that were changed outside Moth.
- Parent paths that are not usable directories.

If a conflict is found, Moth reports it instead of guessing what to delete.

The usual fix is to inspect the target path, move or remove the conflicting file yourself, and run `moth sync` again.

## Obsolete Symlinks

When a previously generated file disappears from the new root `.compiled` tree, Moth removes the old target symlink if it still points to the source recorded in `.targets.local.yaml`.

If the symlink was changed or replaced outside Moth, Moth reports a conflict and leaves it untouched.
